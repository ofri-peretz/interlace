# Color & Theming Philosophy

A focused sub-philosophy of [UX_PHILOSOPHY.md](?path=/docs/philosophy-ux--docs),
paired with [TYPOGRAPHY_PHILOSOPHY.md](?path=/docs/philosophy-typography--docs) —
together they define the page's visual contract. Color is the surface
that, when wrong, telegraphs amateurism faster than any other; when
right, it's the brand recognition signal before the logo loads.

This document binds: what semantic tokens mean, how dark mode is
designed (not derived), the brand-accent rules, the contrast floor,
and the patterns that are explicitly forbidden.

---

## The core rule

> **Color is meaning. Never decoration.** Every color choice maps to
> a semantic role; every semantic role maps to a token; every token
> has a defined contrast budget. If you can't name what a color is
> for, it doesn't ship.

Three corollaries:

1. **Never use color alone to signal state.** Red text without an
   icon is invisible to ~8% of male readers (color blindness).
2. **Tokens, not hex codes.** The hex value can change once, the role
   stays the same.
3. **Dark mode is designed, not derived.** Inverting the palette is
   the wrong call ~80% of the time. Each surface is a separate design.

### Status tokens

| Token | Light | Dark | Use |
| --- | --- | --- | --- |
| `fd-success` | `#15803d` (green-700) | `#4ade80` (green-400) | Pass, complete, available |
| `fd-warning` | `#b45309` (amber-700) | `#fbbf24` (amber-400) | Caveat, deprecated, needs attention |
| `fd-destructive` | `#b91c1c` (red-700) | `#f87171` (red-400) | Error, delete, security alert |
| `fd-info` | `#1d4ed8` (blue-700) | `#60a5fa` (blue-400) | Note, neutral aside, link |

Status tokens are **never decorative**. A red `fd-destructive` border
on a card means "this is destructive," not "let's add color." If a
surface needs visual variety, it uses `fd-accent` or `fd-muted`, not
status colors.

## Brand accent rules

Purple is our accent. That doesn't mean every surface is purple — it
means purple is the signal that *this is the action* or *this is us*.

- **Primary CTAs** — purple background, white text. One per page.
- **Active state in navigation** — purple text on default surface.
- **Focus ring** — purple, 2px, 2px offset.
- **Brand chrome** — logo, badges, highlight strokes.
- **Active filter chip** — purple background with white text.

Where purple is **forbidden**:

- Body text (use `fd-foreground`)
- Most borders (use `fd-border`)
- Backgrounds of large surfaces (use `fd-background` or `fd-muted`)
- Status indicators (use the status tokens, not purple)

The test: if you covered the purple with masking tape, would the page
still be readable? It must be, or the purple is doing the heavy lifting
that contrast and structure should be doing.

## Dark mode contract

Dark mode is a **separate designed surface**. Not "same site, dimmed."

### Rules

- **Backgrounds get progressively lighter as they layer up.**
  - `fd-background` (page) is the darkest layer
  - `fd-card` is one step lighter
  - Modal/popover surfaces another step lighter still
  - The reverse is *wrong* on dark mode (light mode has darker layers
    above lighter; dark mode has lighter layers above darker)
- **Text contrast caps below pure white.** Body text uses `text-zinc-100`
  or `text-zinc-200`, never `text-white`, except for headings.
  Pure white on near-black ghosts on OLED.
- **Status colors shift from saturated → desaturated.** Light-mode
  red-700 becomes dark-mode red-400 — same hue, lighter and less
  saturated. The surface around it is dark; saturated red would
  vibrate.
- **Shadows become glows.** A `shadow-lg` on a card in light mode
  becomes a soft `shadow-purple-500/10` glow on dark mode. Black
  shadows on dark surfaces are invisible.
- **Borders desaturate.** `fd-border` is more visible against dark
  surfaces; reduce opacity to compensate.
- **Images get a subtle treatment** — slight `filter: brightness(0.95)`
  on photos prevents them from punching holes in the dark surface.
- **Brand accent gets brighter.** Purple-600 in light becomes
  purple-400 in dark. Same brand, calibrated for the surface.

### Toggle contract

- **Default to system preference.** `prefers-color-scheme` first,
  user override second.
- **No flash of wrong theme.** Theme set before first paint via a
  blocking inline script, not after hydration.
- **Stored override is per-domain, not per-page.** Reader chose dark
  on `/`, every page is dark.
- **Toggle is reachable.** Keyboard accessible, in a predictable
  location (top-right of the chrome), with `aria-pressed`.

## Contrast budgets

Every text/background combination has a measured contrast ratio.

| Surface | Floor | Target |
| --- | --- | --- |
| Body text | 4.5:1 (WCAG AA) | 7:1 (AAA) |
| Large text (≥ 18px or ≥ 14px bold) | 3:1 (AA) | 4.5:1 (AAA) |
| UI text (button labels, form labels) | 4.5:1 | 4.5:1 |
| De-emphasized text (`fd-muted-foreground`) | 4.5:1 | 4.5:1 |
| Disabled text | 3:1 (AA Large only) | 3:1 |
| Borders separating interactive elements | 3:1 against adjacent surface | 3:1 |
| Focus ring | 3:1 against background AND adjacent border | 3:1 |
| Status icon strokes | 3:1 | 3:1 |
| Decorative imagery | no floor | no floor |

CI enforces the floor on every token combination via an automated
contrast audit (see "Enforcement" below).

## The "color alone" rule

A11y AAA: information conveyed by color is duplicated in another
channel. Concretely:

- **Status badges** carry an icon AND a label AND a color.
  ✓ `<Badge variant="success"><Check /> Passed</Badge>`
  ✗ `<Badge variant="success">Passed</Badge>` (color-only)
  ✗ `<Badge variant="success"><Check /></Badge>` (icon-only)
- **Form errors** show an icon AND a label AND red border.
- **Required fields** show an asterisk OR explicit "(required)"
  text, never just red label color.
- **Links inside body text** carry a color difference AND an
  underline (or non-removable `text-decoration`).
- **Active navigation state** carries a color shift AND a weight
  shift (regular → semibold) AND ideally a structural cue (left
  border, background fill).

## Token aliases & arbitrary values

Tailwind v4 `@theme` block exposes the tokens. **Do not** invent
arbitrary `bg-[#7c3aed]` values — they fragment the palette. Allowed
only for:

- Truly one-off decorative gradients (e.g. cosmic background beams)
- Brand-asset rendering (logo SVGs)
- OG image generation (build-time, not runtime)

Even then: the value is committed to the file as a constant, not
inlined in JSX.

## Theming primitives

Three patterns for how a component picks its colors:

### 1. Semantic-token (default)

```tsx
<div className="bg-fd-card text-fd-card-foreground border border-fd-border" />
```

Component picks tokens for its semantic role. Both light and dark
mode are correct without per-mode classes. Use this everywhere
possible.

### 2. Mode-divergent (when designed differently per mode)

```tsx
<div className="bg-fd-card dark:bg-zinc-900/50 dark:backdrop-blur-md" />
```

When dark mode genuinely needs a different treatment (e.g.
glassmorphism only in dark, drop-shadow only in light). Justify in a
comment.

### 3. CSS-var-driven (when value is dynamic per-instance)

```tsx
<div className="bg-[var(--card-tint)]" style={{ '--card-tint': tint }} />
```

For genuinely runtime-dynamic colors (e.g. tag-color from data).
The CSS-var-via-style is acceptable per `feedback_no_inline_styles`'s
exception clause for runtime values.

## Forbidden patterns

Hard bans.

- **`bg-white` / `bg-black` directly.** Always tokens.
- **Pure CSS named colors** (`red`, `blue`, `green`, etc.) — they're
  inconsistent across browsers and pre-token-era.
- **Random hex codes in JSX.** Tokens or arbitrary-with-justification
  only.
- **Light-text-on-light-bg / dark-on-dark.** Caught by CI contrast
  check.
- **Color-only status signals.** Caught by manual review.
- **`opacity` to dim disabled state.** Use a token specifically for
  disabled (`fd-muted-foreground` + reduced contrast). Opacity also
  dims focus rings.
- **More than 7 hues on one page.** Brand purple, foundation grays,
  4 status colors. Anything else is decoration.
- **Status colors used non-semantically.** Red border for "this is
  cool" is wrong. Red is destructive, not aesthetic.
- **Gradients as primary surface.** Gradients are accents (cosmic
  background, hero chrome), not page backgrounds.
- **Rainbow gradients across `text` / `bg-clip-text` for branding.**
  Dates fast, fails contrast on most stops.

## Enforcement

CI gates that prevent drift:

- **Contrast audit.** Every token pair (foreground × background)
  computed for both modes, fails the build below the floor.
- **Color-only-state lint.** A custom ESLint rule (in our own
  `eslint-plugin-react-a11y`) flags status-colored elements without
  text or icon siblings.
- **Hex-code lint.** A regex rule (`no-raw-color`) bans hex codes in
  JSX outside the allowed-list (the gradient/brand exceptions).
- **Token diff in PR.** Any change to `@theme` is shown as its own
  visual diff in PR (rendered swatches in a comment).

## How this gets used

When designing or reviewing a colored surface, ask:

1. **Token**: which semantic token does this map to? (If you're
   reaching for a hex, stop.)
2. **Mode**: does this surface have a designed treatment in both
   light and dark, or is it derived?
3. **Contrast**: have you measured the foreground × background ratio
   against the budget?
4. **Color-alone**: is the meaning duplicated in icon, label, or
   structure?
5. **Brand restraint**: if you covered the brand purple, would the
   page still be navigable?
6. **Status discipline**: are status colors used only for status?

If any answer is no, the surface is not yet shippable.
